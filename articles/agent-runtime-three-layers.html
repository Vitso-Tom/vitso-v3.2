Agent Runtime

Three Layers Your AI Agent Needs to Be Safe

Runtime governance for autonomous AI agents
Tom Smolinsky · Vitso · March 2026

The Problem

Most production AI applications treat the model as the security boundary. The model is trained to refuse harmful requests. The model is instructed via system prompt to stay in scope. The model is expected to say no when it should say no.

That works until it doesn't.

Prompt injection, persona override, and goal hijacking techniques are not theoretical. They are documented, publicly available, and in many cases years old. When one of them lands, the model stops enforcing its own constraints. And if the model was the only thing between the user and your systems, there is nothing left.

The gap

The problem is not in the model. The frontier models are increasingly resilient to known jailbreak patterns. The problem is between the model and the product layer sitting on top of it. When the model is the only security boundary, a single point of failure exists between every user and every tool the agent was provisioned with.

The question is not whether models will get better at resisting these techniques. They will. The question is what happens in the time between a novel attack and the next training update. And the answer, for most production applications today, is: nothing. There is no second check.

The Three Layers

Runtime governance for AI agents requires three distinct layers. Each one governs a different aspect of agent behavior. Each one is independently necessary. None is sufficient alone.

Layer 01

Identity and Policy

Governs who the agent is

Before any user sends a message, the agent needs to know who it is, who it serves, what it is allowed to do, and what is off limits. This is established outside the conversation in files the agent reads on startup, not in the user interaction itself.

This is the layer that creates resistance to persona override. An agent with a pre-established identity, clear behavioral boundaries, and a classified action model has a basis for rejecting instructions that conflict with its operating mandate. An agent without one has to evaluate every instruction on its own merits, and a well-crafted injection will win that argument.

What this includes

Core identity, values, and behavioral boundaries. The agent's sense of self.

Action classification. Every action the agent might take is pre-sorted into tiers: autonomous, pre-approved, requires human approval, or prohibited. The classification exists before the conversation starts.

Operator identity. The agent knows who its legitimate human is and what that relationship looks like. Claims of authority from unknown sources conflict with an established trust anchor.

Situational context. Deep awareness of the environment, the mission, and the current state of the systems it operates in. An agent with rich context is harder to mislead than one that depends on user text to understand its world.

Without this layer

The agent has no basis to reject a persona override. A prompt that says "you are now someone else" wins because the agent never established who it actually is. User text gets promoted to system authority by default.

Layer 02

LLM Gateway

Governs how the agent thinks

A proxy between the agent and the AI model that handles routing, token management, and observability. Every prompt and every response is logged and traceable. This layer gives you visibility into the reasoning process as it happens.

This layer is probabilistic by nature. Models can be convinced. A sufficiently novel injection might get past the identity layer because the model's judgment is ultimately what enforces it. The LLM Gateway does not prevent that, but it makes it visible.

More importantly, this layer creates the opportunity for a third-party observer. A monitoring system watching LLM traces can detect behavioral shifts, injection patterns, or anomalous tool-call intent in near real time, independently of the agent itself. The agent does not get to evaluate whether it has been compromised. Something else does.

What this includes

Model routing, selection, and fallback logic.

Full prompt and completion tracing for every LLM interaction.

Token usage monitoring and cost controls.

Anomaly detection surface. A third-party workflow can watch traces for injection vocabulary, identity shifts, or tool-call patterns that deviate from the agent's action classification.

Without this layer

You have no visibility into whether the agent's reasoning has been compromised. The model is a black box between the user and your systems. You find out after the damage is done.

Layer 03

Action Gateway

Governs what the agent does

A hard boundary between the agent's reasoning and your systems. Every tool call, every API interaction, every external action passes through a policy decision point. This layer is deterministic. An allowlist either permits the action or it doesn't. A policy engine either approves or denies. The model's internal state is irrelevant.

This is the layer that separates embarrassment from consequence. If the model's identity is overridden, if the persona flips, if the agent starts parroting jailbreak language, the Action Gateway does not care. It does not ask the model whether it believes it should be allowed to call the billing API. It checks the policy. Default posture: ask. Fallback: deny.

What this includes

Action allowlists scoped to specific tools, paths, and operations.

A default posture of "ask" with a fallback of "deny." If the action is not explicitly permitted, it does not happen.

Human approval workflows for elevated actions. The agent stops, requests approval, and waits.

Audit logging of every action attempt, whether approved or denied.

Without this layer

A compromised agent has unrestricted access to every tool and API it was provisioned with at build time. Nobody re-authenticated intent at runtime. Nobody asked whether the model calling the API had been jailbroken. The model was the only security boundary, and a persona override defeated it.

The Distinction That Matters

Two different defenses

Model hardening stops the agent from saying "jailbreak successful." That prevents the embarrassment.

Environment hardening stops the agent from doing the dangerous thing. That prevents the consequence.

Most production AI applications invest heavily in model hardening and stop there. The model resists known attack patterns through training. When a novel pattern lands, or when the model is simply convinced by a well-crafted argument, there is nothing else. The model was the entire defense.

These three layers separate those concerns. Layer 01 helps the model resist the override. Layer 02 gives you visibility into whether the resistance held. Layer 03 blocks the consequence even if both of the above fail. No single layer is the answer. No single layer is dispensable.

With and Without

Without runtime governance

The model is the only security boundary.

User text can promote itself to system authority.

A persona override gives the attacker access to every tool the agent was provisioned with.

The API gateway has no mechanism to evaluate whether the intent behind a call was compromised.

Tools provisioned at build time are never re-authenticated at runtime.

You find out when the damage lands.

With runtime governance

The agent has a pre-established identity that outranks user text.

Every action is classified: autonomous, pre-approved, human-in-the-loop, or prohibited.

A policy decision sits between the model and every tool call. Default is ask. Fallback is deny.

A third-party observer watches the reasoning layer independently of the agent itself.

Intent is re-evaluated at runtime, not assumed from build time.

You find out when the alert fires, not when the damage lands.

Defense in Depth

Security practitioners will recognize this pattern. It is defense in depth applied to AI agent governance. The same principle that puts a firewall, an IDS, network segmentation, and endpoint protection between an attacker and your data. No single control is expected to hold on its own. The system is designed so that the failure of one layer is caught by the next.

The AI agent governance conversation has largely skipped this step. Most frameworks describe what to measure, what to audit, and what to report. They do not describe what to put between a compromised model and your production systems at runtime. These three layers do.

The practical question

If your AI agent was jailbroken right now, what would stop it from using the tools it already has access to? If the answer is "the model would refuse," you have one layer. These are the other two.

This piece covers one application of Agent Runtime: what to put between a compromised model and your production systems at runtime. The broader framework addresses organizational readiness, maturity assessment, action classification, and the transition from human-operated to agent-operated environments. These three layers are the starting point, not the whole picture.
